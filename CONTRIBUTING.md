# Contributing to Awesome AI Engineering

Thanks for your interest in contributing! This list aims to be a high-signal resource for AI engineers building production systems. Quality matters more than coverage.

## What belongs here

A resource belongs on this list if it meets **all** of these criteria:

1. **Genuinely useful to AI engineers.** Think: "Would I recommend this to a colleague shipping an LLM product next week?"
2. **Actively maintained** — for code projects, commits within the last ~12 months.
3. **Reasonably documented** — others can actually adopt it.
4. **Not redundant** — if something equivalent is already listed, add only if yours is materially better for a specific use case.

## What doesn't belong

- Abandoned projects (no commits in 18+ months, unresolved critical issues).
- Pure research code without practical application.
- Proprietary products without clear value beyond marketing.
- Personal blog posts with no lasting reference value.
- Lists of lists (unless exceptionally curated).

## Pull Request Guidelines

### Adding an entry

1. Place the entry in the most appropriate section alphabetically, or in a logical position within the section.
2. Follow this format:

   ```markdown
   - [Name](https://link) — One-sentence description with a period at the end.
   ```

3. Keep descriptions concise — ideally under 80 characters. Describe *what it does*, not *how great it is*. Avoid marketing language.
4. Check for broken links and typos.
5. If adding a new section, include a rationale in your PR description.

### Good vs. bad descriptions

**Good:**
- `[vLLM](https://...) — High-throughput inference engine with PagedAttention.`
- `[Instructor](https://...) — Structured outputs via Pydantic.`

**Bad:**
- `[FooAI](https://...) — The best AI framework ever, super easy to use!`
- `[BarLib](https://...) — A library.`

### Removing entries

If a project is abandoned, superseded, or was added in error, PRs to remove it are welcome. Include a brief justification.

## PR Checklist

Before submitting:

- [ ] My addition follows the format conventions.
- [ ] I've placed the entry in the correct section.
- [ ] The link works and points to the canonical home of the project.
- [ ] The description is neutral and factual.
- [ ] I've checked for duplicates.
- [ ] Commit message is descriptive (e.g., `Add Langfuse to Evaluation & Observability`).

## Code of Conduct

Be respectful, constructive, and assume good faith. Disagreements about whether something belongs are normal; keep them about the work, not the person.

## Questions?

Open an issue before submitting a large PR if you're unsure whether something fits.
